FRAMEWORK / TRELLIS × CSC

把 Trellis 引入 CSC 的工作流治理

Trellis 是「AI 编码的脚手架」——把项目规范、工作流、跨会话记忆自动注入 AI，让 AI 在「规划→执行→收尾」的轨道上工作，而不是凭一次性 prompt 即兴发挥。本页讲它怎么运作、好在哪、以及对 CSC（我们基于 Claude Code 的深度分支产品）意味着什么。

规范注入而非记忆 每轮状态复述 文件优先持久化 子代理分工 一份流程多端运行

核心理念

01
Plan before code

先想清楚再写。规划阶段把需求、约束、方案收敛成文档，再进入编码，避免 AI 边写边猜。

02
Specs injected, not remembered

规范靠注入不靠记忆。每轮把相关规范喂进上下文，而不是寄望模型「记得」几十轮前说过的规矩。

03
Persist everything

research、决策、教训全部落文件。对话会被压缩，文件不会——记忆的真源是磁盘，不是上下文窗口。

Section 1 · 三阶段工作流

Trellis 把任务拆成「规划 → 执行 → 收尾」三段，每段有明确产出物（artifact）。每一步交给带干净上下文的角色去做，不让一个会话从头扛到尾。

1Plan · 规划

brainstorm 阶段：一次只问一个问题，能查就别问，给出选项让人选，先发散再收敛。把模糊需求逼成确定决策。

规划纪律

提问节奏一次一个问题
信息获取能查就别问
决策方式给选项 / 发散→收敛
产出prd.md
2Execute · 执行

实现子代理（trellis-implement）按 prd 写代码；检查子代理（trellis-check）以干净上下文复核。两者分工，各拿只含所需文件的新窗口，互不污染。

双角色分工

实现trellis-implement
检查trellis-check
上下文各自干净窗口
产出implement.jsonlcheck.jsonl
3Finish · 收尾

把这轮学到的东西写回 spec——知识复利：下次同类任务直接吃到这次的经验，而不是重新踩坑。完成后再提交。

知识沉淀

动作学习写回 spec
价值知识复利
收口提交
产出spec/journal

Section 2 · Trellis 真正聪明的设计

这套框架的价值不在「有几个阶段」，而在它对抗长会话失效、上下文漂移的几个工程化决策。下面 6 点是它真正可借鉴的地方。

1

每轮状态面包屑注入

每一轮用户消息都注入「你现在在哪个阶段 + 下一步什么必做」，对抗长会话里指令逐渐衰减、被遗忘的问题。

2

workflow.md 是唯一真源

脚本只是解析器。改流程 = 改一个 markdown 文件，不碰任何代码。流程定义与执行引擎彻底解耦。

3

必做步骤必须每轮复述

血的洞察：required 步骤若不进每轮面包屑，就会被 AI 静默跳过。他们用回归测试守住「required ↔ 面包屑」的对应关系。

4

文件优先的持久记忆

research / prd / spec / journal 全部落在文件里。会话可以被压缩、可以重开，跨会话信息一条都不丢。

5

按任务路由上下文

implement.jsonl / check.jsonl 显式声明每个子代理该读哪些规范——把对的上下文，精确地给对的 agent。

6

子代理分工 + 干净上下文

research / implement / check 各拿只含所需文件的新窗口。实测中 check 子代理跑运行时冒烟，抓出了单测漏掉的崩溃。

Section 3 · 对 CSC 的提升

这是本页最重要的部分。Trellis 验证了「AI 编码需要一个治理层」，但它的实现方式有天花板，而那个天花板恰好是 CSC 能突破的。

核心论点：Trellis 是从外部用 hook / skill「焊」在 Claude Code 上的——所有约束本质上是 prompt 软约束。而 CSC 拥有 harness，能把这些能力从「prompt 软约束」升级为「引擎硬保证」：同样的治理思想，CSC 可以做进引擎，不靠模型自觉。

Trellis 的外挂做法
CSC 可内化的原生能力
每轮用 hook 注入状态
把「每轮状态机复述」做成一等公民，用户不用手写 hook，引擎自动维护。
required ↔ 面包屑靠回归测试守
引擎层强制 required 步骤，不靠 prompt 软约束——结构上跳不过去。
workflow.md 当真源
把「工作流即 markdown / 引擎即解析器」做成平台特性，非工程师也能改流程。
hook 拼 JSONL 注入规范
原生 per-task 上下文路由 API，按任务声明所需规范，引擎负责精准投喂。
Active task: 兜底行（Windows hook 会静默跳过）
改 harness 后注入可靠，从根上省掉这类兜底 hack。
文件优先记忆约定
做成项目级原生记忆模型，记忆是平台能力而非约定俗成。

Section 4 · 务实看待（局限）

客观列出 Trellis 当前的代价与短板，建立可信度。要点在于：这些「软」的地方，恰是 CSC 用 harness 做「硬」的机会。

01

仪式偏重 / token 开销

每轮注入状态与规范，带来固定的上下文开销。Trellis 留了逃生门（可跳过部分仪式），但默认路径偏重。

02

软约束本质

所有约束最终靠模型「听话」执行，因此存在静默跳过的风险——这正是引擎硬保证要解决的核心问题。

03

学习曲线 / 耦合成本

团队需要理解整套工作流与 artifact 约定；markdown ↔ 解析器的耦合也带来持续维护成本。

CONCLUSION / POSITIONING

Trellis 验证了 AI 编码治理层的需求与形态；CSC 拥有 harness，能把它做得更原生、更可靠。

把 Trellis 的治理思想吸收进 CSC：能在 prompt 层做的，CSC 在引擎层做；能靠回归测试守的，CSC 用结构保证。这就是从「外挂脚手架」到「内生平台能力」的跃迁。
