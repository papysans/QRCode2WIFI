FRAMEWORK / TRELLIS × CSC

把 Trellis 引入 CSC 的工作流治理

Trellis 是「AI 编码的脚手架」——把项目规范、工作流、跨会话记忆自动注入 AI，让 AI 在「规划→执行→收尾」的轨道上工作，而不是凭一次性 prompt 即兴发挥。本页讲它怎么运作、好在哪、以及对 CSC（我们基于 Claude Code 的深度分支产品）意味着什么。

规范注入而非记忆 每轮状态复述 文件优先持久化 子代理分工 一份流程多端运行

核心理念

01
Plan before code

先想清楚再写。规划阶段把需求、约束、方案收敛成文档，再进入编码，避免 AI 边写边猜。

02
Specs injected, not remembered

规范靠注入不靠记忆。每轮把相关规范喂进上下文，而不是寄望模型「记得」几十轮前说过的规矩。

03
Persist everything

research、决策、教训全部落文件。对话会被压缩，文件不会——记忆的真源是磁盘，不是上下文窗口。

Section 1 · 三阶段工作流

Trellis 把任务拆成「规划 → 执行 → 收尾」三段，每段有明确产出物（artifact）。每一步交给带干净上下文的角色去做，不让一个会话从头扛到尾。

1Plan · 规划

brainstorm 阶段：一次只问一个问题，能查就别问，给出选项让人选，先发散再收敛。把模糊需求逼成确定决策。

规划纪律

提问节奏一次一个问题
信息获取能查就别问
决策方式给选项 / 发散→收敛
产出prd.md
2Execute · 执行

实现子代理（trellis-implement）按 prd 写代码；检查子代理（trellis-check）以干净上下文复核。两者分工，各拿只含所需文件的新窗口，互不污染。

双角色分工

实现trellis-implement
检查trellis-check
上下文各自干净窗口
产出implement.jsonlcheck.jsonl
3Finish · 收尾

把这轮学到的东西写回 spec——知识复利：下次同类任务直接吃到这次的经验，而不是重新踩坑。完成后再提交。

知识沉淀

动作学习写回 spec
价值知识复利
收口提交
产出spec/journal

Section 2 · Trellis 真正聪明的设计

这套框架的价值不在「有几个阶段」，而在它对抗长会话失效、上下文漂移的几个工程化决策。下面 6 点是它真正可借鉴的地方。

1

每轮状态面包屑注入

每一轮用户消息都注入「你现在在哪个阶段 + 下一步什么必做」，对抗长会话里指令逐渐衰减、被遗忘的问题。

2

workflow.md 是唯一真源

脚本只是解析器。改流程 = 改一个 markdown 文件，不碰任何代码。流程定义与执行引擎彻底解耦。

3

必做步骤必须每轮复述

血的洞察：required 步骤若不进每轮面包屑，就会被 AI 静默跳过。他们用回归测试守住「required ↔ 面包屑」的对应关系。

4

文件优先的持久记忆

research / prd / spec / journal 全部落在文件里。会话可以被压缩、可以重开，跨会话信息一条都不丢。

5

按任务路由上下文

implement.jsonl / check.jsonl 显式声明每个子代理该读哪些规范——把对的上下文，精确地给对的 agent。

6

子代理分工 + 干净上下文

research / implement / check 各拿只含所需文件的新窗口。实测中 check 子代理跑运行时冒烟，抓出了单测漏掉的崩溃。

Section 3 · 聚焦特性 · sub-agent 机制

挑一个最能体现 Trellis 工程巧思的特性细讲：子代理是怎么在「一睁眼」就拿到完整上下文的。关键不在于「派一个子代理」，而在于上下文是怎么被精确装配、并在子代理启动前注入的。

1主会话 dispatch

主会话通过 Task 工具派发子代理（research / implement / check），dispatch 提示词首行写明 Active task: <path>，声明当前活动任务。

触发Task 工具
2PreToolUse hook 拦截

子代理「启动前」被 hook 拦下：解析平台 / 活动任务 → 读 implement.jsonl·check.jsonl 拼出 spec + prd → 重写工具入参 prompt，并打上 <!-- trellis-hook-injected --> 标记。

读取implement.jsonlcheck.jsonl
3子代理一睁眼即满载

子代理被实际拉起时，prompt 已包含完整 spec + prd 上下文，无需任何额外往返就能直接开干——上下文装配发生在它「醒来」之前。

效果零额外往返
4双模式兜底

若 hook 未触发（如某平台不支持 PreToolUse），子代理读 dispatch 首行的 Active task: 自行 pull 上下文。push 失败有 pull 接住。

保障marker + 首行

活样本：本项目里 check 子代理跑运行时冒烟，抓出了「单测全绿却启动崩溃」的 Nest DI bug，并自行修复——干净上下文 + 运行时验证，逮住了纯静态检查漏掉的问题。

1

PreToolUse 改写入参

拦截并重写工具入参注入上下文：行为由代码控制而非 prompt 措辞，确定性强、零往返——不依赖模型「记得」去拉文件。

2

每 agent 一份专属清单

右上下文给右 agent：implement 拿全量 specs + prd，check 拿 check specs，research 拿目录树。jsonl 显式声明，互不串味。

3

角色硬边界 + 递归护栏

research 只描述不评判、implement 禁 commit；「你已是该子代理，别再 spawn」防止子代理无限自我派发。

4

双模式 + 一 hook 通吃

hook push 与 agent pull 互为兜底；同一套 hook 逻辑通吃 8+ 平台编码，不为每个平台各写一份。

Trellis 的外挂做法
CSC 可内化的原生能力
PreToolUse hook 改写入参注入上下文
把「子代理上下文装配」做成原生 API：dispatch 时声明 contextFiles，引擎负责注入，无需 hook 改写入参。
marker + Active task: 双模式兜底
单平台注入确定可靠，从根上砍掉整层 push/pull 兜底 hack。
角色护栏靠 prompt 文字约束
引擎层强制角色边界：禁 commit、禁递归派发由结构保证，不靠模型自觉。
jsonl 显式 per-agent 上下文
把「上下文路由」做成一等概念：声明式描述哪个 agent 吃哪些文件，平台原生投喂。

Section 4 · 聚焦特性 · workflow-state 状态机

再挑一个最能体现「每轮治理」的特性细讲：每条用户消息进来时，Trellis 怎么实时算出「你现在在哪个阶段、这一阶段必做什么」，并把它注入这一轮。关键不在「有个状态」，而在状态的单点写入、纯查表派生，以及让必做步骤每轮都被复述。

1每条消息触发 hook

每一条用户消息进来都触发 hook（Claude Code 上是 UserPromptSubmit，Gemini 上是 BeforeAgent），在模型动笔前先夺过控制权。

触发UserPromptSubmitBeforeAgent
2正则扫 workflow.md

用正则扫 .trellis/workflow.md 里的 [workflow-state:STATUS] 块，得到各状态的文案。workflow.md 是唯一数据源，脚本里没有任何硬编码兜底文案。

读取workflow.md
3拼装并注入这一轮

按 task.json.status 选对应块，拼出 <workflow-state> 注入本轮——头部写当前任务与状态，主体写该阶段必做步骤。

产出<workflow-state>
4状态单点流转

状态流转 no_task → planning → in_progress → completed 由 task.py 生命周期单点写入；面包屑只是 status 的纯查表派生。

真源task.pytask.json

活样本：你正在看的这份对话，每轮顶部的 <workflow-state> 就是它的产出；in_progress 块每轮复述「派子代理 / commit 纪律 / Active task 协议」，正是全程不跑偏的原因。

1

workflow.md 唯一真源 + fail-loud

脚本零兜底字典；某状态的 tag 缺失就输出可见的「Refer to workflow.md」提示，故意让人看见去修，而不是静默掩盖——错误必须吵闹。

2

⭐ required ↔ 面包屑 invariant（血换来的）

每个 [required·once] 步骤必须在该阶段面包屑里有一行强制语句，否则 AI 会静默跳过（曾真实跳过 Phase 1.3 jsonl、3.4 commit）；用回归测试断言这个对应关系。核心真理：没被每轮复述的「必做」等于没有。

3

状态单点写入 + 面包屑纯派生

状态存文件（task.json + .runtime/sessions），面包屑是查表结果；session 隔离，多窗口互不串。

4

no_task 内置 A/B/C 路由

连没任务时也每轮判断：直答 / 建任务 / 带暗号 inline；并用显式暗号短语，防 AI 自己发明例外。

Trellis 的外挂做法
CSC 可内化的原生能力
每轮 hook 注入 status → 面包屑
把「每轮状态机注入」做成一等原语，引擎自动维护，免写 hook。
required 靠约定 + 回归测试守（软）
引擎层保证：required 步骤自动注入提醒 / 缺失即拦截（软约束 → 硬保证，最关键的一条）。
no_task 的 A/B/C 暗号路由
做成可配置的 turn-router 原语，每轮路由声明式可配。
hook 内 14 平台分支（事件名 / Codex notice / dispatch_mode）
单平台砍掉整层兼容复杂度，无需为每个平台各写一套分支。

Section 5 · 对 CSC 的提升

这是本页最重要的部分。Trellis 验证了「AI 编码需要一个治理层」，但它的实现方式有天花板，而那个天花板恰好是 CSC 能突破的。

核心论点：Trellis 是从外部用 hook / skill「焊」在 Claude Code 上的——所有约束本质上是 prompt 软约束。而 CSC 拥有 harness，能把这些能力从「prompt 软约束」升级为「引擎硬保证」：同样的治理思想，CSC 可以做进引擎，不靠模型自觉。

Trellis 的外挂做法
CSC 可内化的原生能力
每轮用 hook 注入状态
把「每轮状态机复述」做成一等公民，用户不用手写 hook，引擎自动维护。
required ↔ 面包屑靠回归测试守
引擎层强制 required 步骤，不靠 prompt 软约束——结构上跳不过去。
workflow.md 当真源
把「工作流即 markdown / 引擎即解析器」做成平台特性，非工程师也能改流程。
hook 拼 JSONL 注入规范
原生 per-task 上下文路由 API，按任务声明所需规范，引擎负责精准投喂。
Active task: 兜底行（Windows hook 会静默跳过）
改 harness 后注入可靠，从根上省掉这类兜底 hack。
文件优先记忆约定
做成项目级原生记忆模型，记忆是平台能力而非约定俗成。

Section 6 · 务实看待（局限）

客观列出 Trellis 当前的代价与短板，建立可信度。要点在于：这些「软」的地方，恰是 CSC 用 harness 做「硬」的机会。

01

仪式偏重 / token 开销

每轮注入状态与规范，带来固定的上下文开销。Trellis 留了逃生门（可跳过部分仪式），但默认路径偏重。

02

软约束本质

所有约束最终靠模型「听话」执行，因此存在静默跳过的风险——这正是引擎硬保证要解决的核心问题。

03

学习曲线 / 耦合成本

团队需要理解整套工作流与 artifact 约定；markdown ↔ 解析器的耦合也带来持续维护成本。

CONCLUSION / POSITIONING

Trellis 验证了 AI 编码治理层的需求与形态；CSC 拥有 harness，能把它做得更原生、更可靠。

把 Trellis 的治理思想吸收进 CSC：能在 prompt 层做的，CSC 在引擎层做；能靠回归测试守的，CSC 用结构保证。这就是从「外挂脚手架」到「内生平台能力」的跃迁。
